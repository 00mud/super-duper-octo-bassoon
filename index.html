01. Native Sync Services

Native solutions prioritize resource conservation. By employing a "Pull" logic, the browser only checks for updates at intervals. While sufficient for data redundancy (backup), it fails to facilitate live "handoff" scenarios where a user requires immediate access to a session on a secondary device.

02. Vertical Organizers

Extensions like Sidebery revolutionize Spatial HCI (Human-Computer Interaction). They solve the "Tab Overload" problem vertically. However, they lack a networked state reconciliation engine, meaning the organization remains trapped on a single machine.

03. Session Archivers

Tools like OneTab focus on Memory Optimization. They suspend tabs into static lists to save RAM. While effective for storage, they do not support the active, bi-directional state mirroring required for modern multi-device workflows.

Architectural Innovation: Push vs. Pull

Our literature review identified the Push-Response bottleneck as the primary research gap. Most systems wait for a request before delivering data.

Tabbed-In utilizes the WebSocket (RFC 6455) protocol. Unlike HTTP polling, WebSockets provide a full-duplex communication channel. When Device A triggers an onCreated event, the server "pushes" that state to all listeners instantly, removing the client-side wait time entirely.

System Methodology

The Logic Architecture of Tabbed-In

1. Event Interception

Background scripts monitor onUpdated and onRemoved hooks. When a tab state changes, the extension captures the URL, Title, and unique Window ID.

2. State Serialization

Captured data is serialized into a JSON payload including a timestamp and origin_device_id. This is broadcasted via Supabase WebSockets using a "Push" pattern.

3. State Reconciliation

Remote devices receive the payload and compare it against their local browser.tabs.query results. It triggers execution only if a delta (difference) is detected to prevent infinite loops.

Concurrency Handling

To prevent Race Conditions where two devices update the same tab simultaneously, Tabbed-In utilizes a Last-Write-Wins (LWW) approach. Each broadcast carries a high-resolution UTC timestamp; receiving clients only execute updates if the incoming timestamp is newer than the last known local state for that specific tab ID.

The Persistence Layer

While Realtime handles the active "shouting" of updates, the Session Snapshot feature uses PostgreSQL transactions. This ensures that even if a device is offline, it can fetch the latest "Mirrored" state from the database upon reconnection, reconciling the missed events during the offline period.
